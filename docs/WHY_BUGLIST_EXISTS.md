# 为什么会有 Buglist

## 一套面向真实 UI 场景的 AI 原生自动化测试系统说明

---

## 1. 文档目的

这份文档不是单纯介绍 `Buglist` 有哪些功能，而是回答更重要的几个问题：

1. 为什么需要做这样一套系统
2. 传统自动化测试方案到底卡在哪里
3. 这套系统和传统方案相比，有什么本质差异
4. 它具体解决了哪些过去难以解决的问题
5. 它适合被放在什么样的团队和业务场景中

如果一句话概括：

`Buglist` 的出现，是为了让自动化测试从“依赖固定规则的脆弱脚本”，升级成“由 AI 理解目标、由浏览器执行动作、由视觉判断结果、并最终闭环到报告与缺陷系统”的智能测试平台。

---

## 2. 背景问题：为什么传统自动化测试越来越难维护

在很多团队里，自动化测试并不是不会写，而是“写得出来，维护不起”。

尤其是 UI 自动化，随着页面越来越复杂、前端框架越来越动态、业务变更越来越频繁，传统测试脚本会遇到几个非常典型的问题。

### 2.1 过度依赖 DOM、CSS 选择器和页面结构

传统自动化测试最常见的写法是：

- 用 XPath 找元素
- 用 CSS Selector 找元素
- 用固定文本或类名定位

这种方式在页面结构稳定时非常有效，但一旦页面发生下面这些变化，就会快速失效：

- className 重构
- DOM 层级调整
- 前端组件重排
- 弹窗层级变化
- 按钮位置变化
- 同类元素数量变化

结果就是：

- 脚本本身没有错
- 页面功能也没有错
- 但定位规则失效了
- 测试却报失败

这类失败不是业务 bug，而是自动化脚本自身的脆弱性。

### 2.2 很多测试需求天然是“自然语言”，不是“规则语言”

真实测试需求经常这样描述：

- 点击顶部报名按钮
- 点击右侧可用 USDT 右边的加号
- 打开分享弹窗后点击 Twitter 图标
- 报名成功后按钮应显示为“已报名”
- 验证页面是否真的跳转到了 Telegram 分享页

这些描述对于人来说很好理解，但要把它们全部翻译成稳定的选择器和断言规则，工作量很大，而且容易写错。

也就是说，**测试需求本身是语义化的，而传统执行器是规则化的**，两者之间存在天然鸿沟。

### 2.3 页面结果判定往往比页面操作更难

自动化测试经常不是“点不到”，而是“点完以后不知道到底算不算成功”。

例如下面这些断言：

- 按钮是否真的变成了“已报名”
- 弹窗是否真的关闭
- 当前页面是不是 Twitter 页面
- 页面上的文案变化是不是出现在目标控件本身
- 页面是不是被通知弹窗遮挡了

如果只靠 DOM 文本去判断，很容易出现误判：

- 页面别处也出现了“已报名”三个字
- 但目标按钮其实没变
- 脚本却判定通过

这类问题在传统自动化里非常常见，因为“规则断言”不擅长处理“视觉语义”。

### 2.4 登录态、弹窗、新标签页、外链跳转让脚本更脆弱

真实业务页面远比 demo 页面复杂。

它们可能包含：

- 登录态限制
- 新标签页跳转
- 多层弹窗
- 分享弹层
- 浏览器权限弹窗
- 页面遮挡层
- 外部社交平台分享页面

这些都意味着：

- 不是单纯写一个 `click()` 就完了
- 也不是单纯写一个 `expect(text)` 就够了

传统脚本在这些场景里通常要写大量特判、分支和兜底逻辑，维护成本快速上升。

### 2.5 执行和缺陷提报是割裂的

很多团队即使做了自动化测试，后续流程仍然是人工：

1. 自动化工具告诉你失败了
2. 人去看截图
3. 人去整理原因
4. 人去提 bug
5. 人再把截图和步骤贴到缺陷系统

这说明整个流程并没有闭环，自动化只做到了“执行”，并没有真正做到“交付结果”。

## 3. 传统方案的局限性

这一部分可以把常见方案拆开看。

### 3.1 纯 Playwright / Selenium 脚本方案

优点：

- 控制浏览器能力强
- 执行速度快
- 工程化成熟
- 适合结构稳定的页面

缺点：

- 非常依赖选择器质量
- 面对复杂语义目标时很难写
- 对视觉结果理解能力弱
- 容易因为页面小改动导致大量脚本报错

结论：

它擅长“执行”，但不擅长“理解”。

### 3.2 录制回放类方案

优点：

- 上手快
- 初期出结果快

缺点：

- 对页面变化极其敏感
- 很难覆盖复杂业务场景
- 难以表达抽象测试意图
- 经常录得出来但复用不了

结论：

它适合演示，不适合真实长期维护。

### 3.3 纯 OCR / 纯文本规则方案

优点：

- 可做一定程度的文案识别
- 比纯 DOM 多了一层感知能力

缺点：

- 不理解页面结构关系
- 不擅长判断“这个文字属于哪个控件”
- 容易误把页面其他位置的相同文案当成成功

结论：

它能读字，但不真正理解界面。

### 3.4 纯 Computer Use 方案

优点：

- 像人一样操作电脑
- 不强依赖 DOM
- 泛化能力强

缺点：

- 速度慢
- 成本高
- 可重复性不稳定
- 对大量回归测试不够友好

结论：

它擅长“像人”，但不适合大规模高频执行。

## 4. 为什么 Buglist 要采用现在这套设计

`Buglist` 的核心设计思路并不是否定传统自动化，而是重新分工：

- **AI 负责理解**
- **Playwright 负责执行**
- **AI 负责判定**
- **系统负责闭环**

也就是说，它不是完全抛弃脚本引擎，而是把传统脚本里最脆弱的部分替换成 AI 能力。

这套设计的核心判断是：

### 4.1 浏览器操作应该交给稳定的自动化引擎

像下面这些动作，Playwright 非常擅长：

- 打开页面
- 点击元素
- 输入内容
- 等待页面变化
- 切换上下文

所以执行层不需要重新发明轮子。

### 4.2 语义理解应该交给 AI

像下面这些问题，AI 比硬编码规则更合适：

- “顶部报名按钮”到底指哪个元素
- “右侧可用 USDT 右边的加号按钮”该怎么理解
- “当前页面是不是 Telegram 分享页”
- “按钮状态是否已经变成已报名”

这些问题本质上都不是浏览器问题，而是理解问题。

### 4.3 结果判断不能只看规则，要看页面最终状态

这是 `Buglist` 和很多自动化工具最大的区别之一。

在这套系统里，判定不是停留在：

- click 成功了没有
- 某个文本存不存在

而是进一步强调：

- 目标控件本身状态有没有变化
- 当前页面最终状态是不是符合 `expected_result`

所以系统引入了：

- 单步骤 AI 视觉断言
- 页面级目标验证工具
- 用例级最终终态校验

这三层一起工作，能显著降低“机械执行成功但业务结果其实错误”的误判风险。

## 5. Buglist 当前系统的核心差异

### 5.1 它不是“AI 生成测试用例工具”

很多产品只把 AI 用在前半段：

- 帮你生成测试用例
- 帮你写点断言建议

但真正执行时，仍然是传统规则引擎。

`Buglist` 不一样，它把 AI 放进了执行器里面。

也就是说，AI 不只是“生成器”，还是“执行过程中的参与者”。

### 5.2 它不是“纯浏览器脚本工具”

传统浏览器脚本引擎擅长执行，但不会理解测试语义。

`Buglist` 当前执行链路里，AI 会参与：

- 步骤语义解析
- 点击目标定位
- 是否切页判断
- 页面级目标验证
- 普通视觉判定
- 最终终态校验

这意味着引擎不是“先写死再执行”，而是“边理解边执行”。

### 5.3 它不是“只看一条规则就给结论”

对于某些复杂场景，例如：

- Twitter / Telegram / Facebook 分享跳转
- 目标页面验证
- 状态按钮变化

系统不是只看一项信号，而是综合：

- 截图
- URL
- 标题
- 页面正文摘要
- 当前步骤语义
- 用例预期结果

这使得结论更接近人工测试员的判断逻辑。

## 6. 这套系统解决了什么问题

下面按问题类型来说明它的价值。

### 6.1 解决“自然语言测试步骤很难落地”的问题

过去：

- 测试人员写的是人话
- 自动化引擎需要的是选择器和硬编码逻辑

现在：

- 测试步骤可以保持自然语言
- AI 负责把自然语言标准化成可执行语义

这降低了从“测试意图”到“可执行步骤”的转换成本。

### 6.2 解决“页面结构变化导致脚本大量失效”的问题

过去：

- 页面稍改，XPath/CSS 规则就挂

现在：

- AI 先理解要找什么
- 再从候选元素中选择最符合语义的目标

这让元素定位从“死规则”升级为“语义匹配”。

### 6.3 解决“复杂跳转场景很难验证”的问题

过去：

- 打开新标签页后很容易丢上下文
- 很难确定当前打开的是不是目标页面

现在：

- AI 决定是否需要切换新标签页
- 页面级工具负责判断当前是否已到达目标页

这让外部分享、社交页面跳转等场景更稳定。

### 6.4 解决“视觉状态断言容易误判”的问题

过去：

- 只看文字，容易误判
- 页面别处有相同文案，也可能被算成功

现在：

- AI 根据截图理解目标控件本身
- 最后再结合整条用例的 `expected_result` 做最终终态校验

这让系统更接近人工测试，而不是机械匹配文本。

### 6.5 解决“登录页面、活动页、弹窗页难自动化”的问题

过去：

- 这些页面经常有遮挡、跳转、登录态依赖

现在：

- 支持专用浏览器登录态复用
- 支持分享弹窗类特殊验证
- 支持页面最终状态视觉确认

这让系统更适合真实业务，不只是 demo 页面。

### 6.6 解决“测试结果和缺陷系统脱节”的问题

过去：

- 自动化执行失败后，还要人工整理 bug

现在：

- 平台能自动收敛失败结果
- 自动保留关键截图
- 自动整理 AI 判定说明
- 一键提交到禅道

这把测试执行和缺陷闭环连接起来了。

## 7. Buglist 相比传统方案的对比

| 对比维度 | 传统自动化 | Buglist |
| --- | --- | --- |
| 步骤表达 | 需要人工转成规则 | 可直接使用自然语言测试步骤 |
| 元素定位 | 依赖 XPath / CSS / DOM | AI 语义定位 + 候选元素选择 |
| 跳转场景 | 新标签页容易丢失上下文 | AI 决定切页并验证目标页 |
| 结果判断 | 多依赖文本或规则断言 | AI 视觉判定 + 页面级验证 |
| 最终结果可靠性 | 容易出现假通过 / 假失败 | 有单步骤校验和最终终态校验 |
| 登录场景 | 需要单独处理复杂流程 | 支持专用浏览器登录态复用 |
| 报告输出 | 常常只有 pass / fail | 带 AI reason、截图、失败步骤 |
| 缺陷闭环 | 往往手工提交 | 可一键提交禅道 |

## 8. 这套系统最适合哪里

### 8.1 高变化页面

例如：

- 活动页
- 营销页
- 分享页
- 交易页

这类页面 DOM 变化频繁、控件位置变化大，最适合引入语义定位和视觉断言。

### 8.2 需要快速验证需求的团队

当团队需求变化快、测试用例写得快、页面也改得快时，传统自动化维护成本会很高。

`Buglist` 更适合作为这种场景下的“高灵活度自动化测试引擎”。

### 8.3 需要展示 AI 与工程能力结合的场景

例如：

- 黑客松
- PoC
- 内部技术分享
- 新型测试平台实验项目

它非常适合拿来证明：

“AI 不是只能生成文字，也可以直接参与复杂工程流程。”

## 9. 当前系统的意义

`Buglist` 当前最重要的意义，不只是“多了一个测试工具”，而是验证了一种新的测试执行范式：

### 过去的范式

测试意图  
-> 人工写规则  
-> 浏览器执行  
-> 人工判断结果  
-> 人工提 bug

### Buglist 的范式

测试意图  
-> AI 理解步骤  
-> 浏览器执行  
-> AI 理解页面结果  
-> 系统输出报告  
-> 系统闭环缺陷提交

这就是它和传统自动化测试最大的不同。

---

## 10. 一句话总结

为什么会有这个系统？

因为传统自动化测试最大的问题，不是“不能执行浏览器”，而是“不能像人一样理解页面和结果”。

`Buglist` 的出现，就是为了把自动化测试从“规则驱动”推进到“语义驱动 + 视觉驱动 + 浏览器执行 + 结果闭环”的下一阶段。

如果用一句更适合演示的话来说：

**Buglist 不是在替代测试人员写脚本，而是在让自动化测试第一次真正具备接近人工测试的理解能力。**
